## Overview

The **Activity Report per-event table** is a two-column markdown table rendered by
`build_event_timeline_table(raw_event_log, config, hours, now)` in
`ai_skills_activity.py`. It is **deterministic Python** — not generated by the LLM.

After `parse_activity_response()` parses the Claude response, the coordinator
**overrides** `sections["timeline"]` with the output of `build_event_timeline_table`.
The LLM is responsible only for the `summary`, `decisions`, and `anomalies` sections;
it never writes the timeline table.

The table also replaces the timeline in `activity_fallback()`, so the deterministic
rendering applies even when Claude is unavailable.

**Column semantics:**

| Column | What it contains |
|---|---|
| **Time** | Local HH:MM of the event |
| **Event** | What happened — a human-readable label for the trigger or action type |
| **Settings** | The concrete HVAC action performed — setpoint, mode change, fan state, or skip reason |
| **Indoor** | Indoor temperature at the moment the event was emitted, from `coordinator._get_indoor_temp()`. Displays `—` for events recorded before Issue #352 enrichment. |
| **Outdoor** | Outdoor temperature at the moment the event was emitted, from `coordinator._last_outdoor_temp`. Same availability caveat. |

The split between Event and Settings lets users scan the Settings column to understand
_what the thermostat was told to do_ independently from the event labels. Indoor and
Outdoor make the ambient conditions visible alongside each decision, eliminating the
need to cross-reference chart data to understand why a setpoint was applied.

## Setpoint Convention

Climate Advisor operates **single-setpoint** (Issue #301). At any moment the
thermostat holds exactly one active setpoint; the other comfort edge is the monitored
bound that would trigger the next action.

`_format_band_setpoint(floor, ceiling, active, unit)` renders both values:

| `active` value | Rendered string |
|---|---|
| `"ceiling"` | `setpoint: {ceiling} Cool ({floor} Heat)` |
| `"floor"` | `setpoint: {floor} Heat ({ceiling} Cool)` |

- The **first value** is the live thermostat setpoint (`climate.set_temperature` was
  called with this value).
- The **parenthesised value** is the monitored bound — the edge that, if crossed, would
  trigger the next action.

**Examples:**

- `active="ceiling"`, `ceiling=72°F`, `floor=64°F` →
  `setpoint: 72°F Cool (64°F Heat)`
- `active="floor"`, `floor=64°F`, `ceiling=72°F` →
  `setpoint: 64°F Heat (72°F Cool)`

Temperature values are formatted by `temperature.format_temp(value, unit)` to respect
the user's configured unit (°F or °C).

One row per emitted event type. Emitters are `automation.py` (via
`_emit_event_callback`) and `coordinator.py` (via `_emit_event`).

## Default Renderer

Any event type **not registered** in `EVENT_RENDERERS` is handled by
`_default_renderer(event_type, data)`:

- **Event column:** humanized type string (underscores → spaces, title-case) + `reason`
  field appended if present. Example: `"nat_vent_fan_on"` → `"Nat Vent Fan On"`.
- **Settings column:** generic extraction of recognized fields in this priority order:
  `old_setpoint_f`/`new_setpoint_f` → `setpoint: {old}→{new}`;
  `old_hvac_mode`/`new_hvac_mode` → `mode: {old}→{new}`;
  `floor`/`ceiling`/`active` → `_format_band_setpoint(...)`;
  `fan` → `fan: {fan}`;
  `trigger` → `trigger: {trigger}`;
  else blank.

**The default renderer never raises and never produces a blank Event column.** This
means a newly emitted event type renders meaningfully with zero code changes. Adding a
tailored renderer to `EVENT_RENDERERS` is opt-in.

## Dedup Contract

Consecutive rows with the **same event type** are collapsed into a single row:

```
{friendly name} ×{N} ({first_HH:MM}–{last_HH:MM})
```

**The Settings cell is preserved** — all collapsed rows share the same band parameters
(they represent the same comfort state being re-armed), so the Settings text from the
first (or last) instance is used verbatim.

**Indoor and Outdoor cells use the first event in the collapsed run.** Temperature
conditions at the start of a repeated band cycle are more informative than the last.

This fixes the historical defect where `Sleep comfort band applied ×18` showed an empty
Settings column (Issue #330): the collapse preserves the `_format_band_setpoint` output
from the band's payload.

**Collapse applies only to consecutive same-type rows.** If event type A appears, then B
appears, then A appears again, the two A rows are not collapsed — they represent distinct
occurrences.

## Runbook: Adding or Changing a Renderer

Follow these four steps any time you add a new event emission or want to improve an
existing renderer.

### Step 1 — Emit with structured fields (not prose)

The event payload must carry all values needed to render the Settings column as discrete
fields. Do **not** embed the Settings text in the `reason` string — that reverts to
LLM-dependent parsing.

```python
# Good: structured
self._emit_event_callback(
    "my_new_event",
    {"setpoint_f": 72.0, "mode": "cool", "trigger": "user_schedule"},
)

# Bad: prose in reason
self._emit_event_callback(
    "my_new_event",
    {"reason": "set to 72F cool because user schedule"},
)
```

### Step 2 — Add an `EVENT_RENDERERS` entry (optional but recommended)

In `ai_skills_activity.py`, add an entry to `EVENT_RENDERERS`:

```python
EVENT_RENDERERS: dict[str, Callable[[dict, str], tuple[str, str]]] = {
    ...
    "my_new_event": lambda data, unit: (
        f"My new event — {data.get('trigger', 'unknown')}",
        f"setpoint: {format_temp(data['setpoint_f'], unit)} {data['mode'].title()}",
    ),
}
```

If you omit this step, `_default_renderer` handles the type automatically. The renderer
is optional but provides a precise, stable display. Add it if the event is high-frequency
or user-visible.

### Step 3 — Add a catalog row to this document

Add a row to the appropriate group table in the [Event Catalog](#event-catalog) above:

```
| `my_new_event` | `my_emitter_file.py` `my_function` | `setpoint_f`, `mode`, `trigger` | "My new event — {trigger}" | `setpoint: {setpoint_f} {mode}` |
```

This is the single source of truth. The catalog row and the `EVENT_RENDERERS` entry must
agree on field names.

### Step 4 — Add a unit test case

In `tests/test_ai_activity_table.py`, add a test that asserts:
- The Settings column is non-empty for your new event.
- The rendered text matches the expected format.

```python
def test_my_new_event_settings_populated():
    table = build_event_timeline_table(
        [{"time": "10:00", "type": "my_new_event",
          "setpoint_f": 72.0, "mode": "cool", "trigger": "user_schedule"}],
        config={"temp_unit": "fahrenheit"},
        hours=24,
        now=datetime.now(),
    )
    assert "72" in table
    assert "cool" in table.lower()
```

## Coverage Guardrail

`tests/test_ai_activity_table.py` contains a **coverage-guardrail test** that:

1. Introspects `automation.py` and `coordinator.py` source text for every
   `_emit_event(_callback)?("type"` literal using a regex.
2. Asserts each discovered type is either:
   - present as a key in `EVENT_RENDERERS`, **or**
   - listed in `INTENTIONAL_DEFAULT_TYPES` — a module-level set in
     `ai_skills_activity.py` for types that intentionally use `_default_renderer`.

If a developer adds a new `_emit_event_callback("my_new_type", ...)` call without
updating `EVENT_RENDERERS` or `INTENTIONAL_DEFAULT_TYPES`, this test fails loudly.

The guardrail also asserts that `_default_renderer` produces:
- A non-empty Event string for a synthetic unknown type.
- A non-crashing Settings string for a payload with arbitrary fields.

**This test exists to prevent the recurring defect (Issues #206, #216, #290, #329, #330)
where new event types produced blank or LLM-dependent rows in the activity table.**
